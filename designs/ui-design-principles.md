# UI 設計方針 — Nostr Deck

このドキュメントは UI の**設計哲学と原則**を言語化したもの。個々の画面仕様ではなく「なぜそう作るか」の判断基準を示し、迷ったときの拠り所にする。
実装の全体像は [whiteboard.md](../whiteboard.md)、動くモックは [index.html](./index.html)、色/寸法の値は [tokens.css](./tokens.css)。

---

## 0. 一行で言うと

**「畳めば普通の SNS、開けば本領の Deck」** — 1つの画面を幅で連続的に変形させ、フォルダブル/タブレットの広さを"展開時の本質"として活かす。

---

## 1. 核となる原則（Principles）

### P1. Deck ファースト、しかし押し付けない
Deck（複数カラム）がこのアプリの本質だが、狭い画面で無理に見せない。**狭い＝1カラムの素直な SNS、広い＝Deck**。ユーザーは「広げたら情報が増えて嬉しい」を体験する。Deck を理解していなくても狭い画面だけで完結して使える。

### P2. 幅で駆動する（ヒンジで駆動しない）
レイアウト分岐は**ウィンドウ幅**（WindowSizeClass）で決める。折りたたみの開閉・iPad の Split View / Stage Manager・PC ウィンドウのリサイズを**すべて同じ経路**で扱える。ヒンジ検出はガター（折り目回避の余白）という装飾にのみ使い、ヒンジ API の無い iOS でも破綻させない。

### P3. すべては「カラム」、ピンで永続化
フィード・スレッド・チャンネルルームを区別なく `ColumnSpec` として扱う。違いは**レンダラー**と**ピン状態**だけ。これにより「開く・固定する・並べ替える・閉じる」という操作語彙が全コンテンツで一貫する。新しいコンテンツ種別が来ても同じ枠に乗せる。

### P4. UI はネットワークを見ない（オフラインファースト）
画面はローカル DB だけを読む。回線状態は描画の前提条件にしない。読み込み中・オフライン・同期保留は**状態の表現**であって、エラーや行き止まりにしない。常に「最後に見えていたもの」が即出る。

### P5. ネイティブの軽さを体感に変える
固定幅カラム＋オフスクリーン非描画、imeta/blurhash の即時プレースホルダ、安定キーによる最小再描画。**「速い」を機能として設計**する（フレームワーク任せにしない）。

### P6. 自分の身元は自分のもの
npub/nsec・鍵・リレー選択をユーザーが握っている感覚を UI で支える。プラットフォーム的な「お任せ」ではなく、**何がどのリレーから来て、どこへ出るか**を必要に応じて可視化できる。

## 2. レイアウトの大方針

### 2.1 固定幅カラム + はみ出しは横スクロール
- カラム幅は固定（340dp 基準, `tokens: --column-w`）。画面幅で**見える枚数だけ**が変わり、残りは横スクロール。
- 理由：可変フィット（幅で均等割り）より軽く、どんな幅でも同じロジックで成立する。iPad の連続リサイズ・将来の Wide 端末にそのまま効く。

### 2.2 幅のブレークポイント
| 幅 | モード | レイアウト |
|---|---|---|
| Compact（〜600dp） | 折りたたみ・縦持ちスマホ | 1カラム + 下部ナビ + タブ/Pager |
| Medium/Expanded（600dp〜） | 展開・タブレット・PC | 左レール + 横スクロール Deck |

### 2.3 折り目（ヒンジ）への配慮
展開時、ヒンジ位置にカラム境界が重なるようガター（`--hinge-gutter`）を挿入し、1カラムが折り目で割れないようにする。テーブルトップ姿勢（半開き）では上下分割レイアウトの余地を残す。

## 3. ナビゲーション方針（アダプティブ）

### 3.1 器を幅で入れ替える
「広げたらナビが消える」を作らない。**下部 NavigationBar（Compact）⇄ 左 NavigationRail（Expanded）** と器ごと昇格させる（Material 3 アダプティブ準拠）。

### 3.2 役割の住み分け
- **コンテンツ＝カラム**：Following・ハッシュタグ・リスト・チャンネル・スレッド。Deck に並ぶ。
- **アプリ機能＝ナビ宛先**：Home・検索・通知・DM・設定。レール上部／下タブに常駐。

### 3.3 左レールは「目次」
レールは2層：上＝グローバルナビ宛先、下＝**ピン留めカラムのアイコン一覧**。レールのアイコンをタップすると Deck をそのカラムへ `animateScrollTo` でジャンプ（＝一瞬の切替）。ドラッグで並べ替え、未読はバッジ。

---

## 4. カラムの統合モデル

### 4.1 2つの状態
- **transient（一時）**：タップで開き ✕ で閉じる。実行時のみ。例：ノート→スレッド、チャンネル→ルーム。
- **pinned（固定）**：永続化＋レール常駐＋並べ替え可。例：Following・お気に入りチャンネル・残したいスレッド。
- **ピン留め＝transient を pinned へ昇格**。逆も可。これが「好きなものをカラム一覧に固定」の実体。

### 4.2 3つのレンダラー（同じカラムでも見せ方を変える）
| レンダラー | 対象 | 並び | 入力欄 | 体験 |
|---|---|---|---|---|
| FEED | kind:1（Following/hashtag） | 逆時系列 | なし（FAB） | 読み物 |
| THREAD | NIP-10 返信ツリー | ツリー（root→対象→返信） | 返信ボックス | 文脈を追う |
| ROOM | NIP-28 kind:42 | **時系列昇順・最新が下** | **下部常設** | チャット |

「読み物（逆時系列）」と「チャット（昇順・下入力）」を**取り違えない**のが最重要。並び順と入力欄の位置で種別を直感させる。

### 4.3 一貫した操作語彙
全カラムで「開く／固定する／並べ替える／閉じる／設定（=フィルタ編集）」を共通化。カラム設定は `{kind, authors, #t, relays...}` の編集 UI として表現する。

## 5. オフライン/同期の UI 表現

- **stale-while-revalidate**：まずキャッシュを即描画、裏で更新。更新中もコンテンツは消さない。
- **スケルトン**：未取得部分のみ。画面全体のスピナーで止めない。
- **プレースホルダ即時化**：画像は imeta/blurhash で寸法確定 → スクロールがガタつかない（CLS 防止）。
- **オフラインは状態であってエラーでない**：控えめなバナー（同期保留件数など）で示し、操作はブロックしない。
- **楽観的反映**：自分の投稿/リアクションは即 UI に出し、送信は裏でキュー。

## 6. ビジュアルアイデンティティ（差別化）

### 6.1 Damus との差別化
差別化はノート単体の装飾ではなく**構造**で出す：**Deck × 左レール × カラムで開くスレッド/チャット**。単一カラム・スタック遷移のクライアント群とは情報の動き方が違う。

### 6.2 トーン
- ダーク基調（Nostr クライアントの文脈に合わせる）、アクセントは Nostr パープル。
- 情報密度は中〜高（Deck は一覧性が価値）。ただし行間と区切り線で**詰め込みすぎを回避**。
- アクションはアイコン＋数値、色は意味づけ（zap=ゴールド/repost=緑/like=ピンク/reply=パープル）。

### 6.3 トークン同期
色・寸法は `tokens.css` と Compose の `theme/Color.kt` `Theme.kt` を**一対一**で保つ。デザインモックと実装が乖離しないことを仕組みで担保する。

## 7. インタラクション原則

- **タップは文脈を増やす方向**：ノート→右にスレッド、チャンネル→右にルーム。戻るより「閉じる」。Deck の空間に積み上げる感覚。
- **ジャンプは即時**：レールアイコン→該当カラムへスクロール。階層を潜らせない。
- **破壊的操作は明示**：カラム削除・固定解除は取り消し可能な形（透明性）。
- **長押し/ドラッグ**：並べ替え。ジェスチャは補助であり、すべてタップでも到達可能にする。

---

## 8. アクセシビリティ & フォーム対応

- 可変幅前提なので**最小タップ領域 48dp**を死守（カラムが細くても崩さない）。
- フォントスケール拡大で破綻しない（固定高セルを避ける）。
- 片手操作：主要操作は下方（FAB・下タブ・ルームの下部入力）に寄せる。
- フォルダブル姿勢（テーブルトップ）を将来の入力レイアウト最適化の余地として残す。

---

## 9. やらないこと（Non-goals）

- 幅ごとに**別々の画面**を作る（＝二重メンテ）。あくまで1つの可変レイアウト。
- 狭い画面に Deck を無理押し込み（横スクロール地獄）。Compact は1カラムに徹する。
- プラットフォーム標準リスト装飾への依存（カスタム Deck なので CMP/Skia 描画で完結させる）。
- ネットワーク状態を前提にした画面遷移（オフラインで行き止まりを作らない）。

---

## 付録：この方針が効く判断例
- 「タブレットでナビが消えるのでは？」→ P2/3.1：下タブを左レールに昇格。
- 「スレッドはどう見せる？」→ P3/4.2：右に THREAD カラムで開く（スタック遷移しない）。
- 「パブリックチャットの置き場は？」→ P3/4：一覧は list 宛先から開くカラム、ルームは ROOM レンダラー、固定はピン。
- 「回線が不安定」→ P4/5：キャッシュ即描画＋保留バナー、エラーにしない。
